# ADR-006 — Intent Engine Architecture

Date: 2026-07-16

Status: Accepted

---

## Context

The current simulation pipeline skips the **Intent** stage documented in Architecture/04 and Architecture/05.

Deliveries are generated by applying player-strength modifiers directly to a base probability distribution. There is no concept of what a batter or bowler *intends* to do on any given delivery.

This produces flat, context-blind simulation. A batter with batting=75 behaves identically whether the score is 20/0 in over 3 or 155/7 in over 19. A bowler does not change their approach between powerplay and death overs.

The documented simulation pipeline explicitly separates intent from probability:

```
Context → Intent → Probability → Outcome → State
```

Intent is the missing layer between context and probability.

---

## Decision

Introduce an **Intent Engine** as a first-class component of the delivery pipeline.

The Intent Engine produces a `DeliveryIntent` object before probability calculation begins.

```
IntentEngine.resolve(context: MatchContext, striker: Player, bowler: Player): DeliveryIntent
```

`DeliveryIntent` carries:
- `BatterIntent` — DEFEND / ROTATE / ATTACK / ACCELERATE
- `BowlerIntent` — CONTAIN / ATTACK / YORKER / BOUNCER

Intent drives the `BatterIntentModifier` and `BowlerIntentModifier` within the Probability Engine.

---

## Intent Resolution Rules

### Batter Intent

Intent is determined primarily by innings phase and match situation.

| Situation | Intent |
|---|---|
| POWERPLAY, established batter | ATTACK |
| DEATH, 1st innings | ACCELERATE |
| DEATH, 2nd innings, RRR > 12 | ACCELERATE |
| DEATH, 2nd innings, RRR 8–12 | ATTACK |
| DEATH, 2nd innings, RRR < 8 | ROTATE |
| MIDDLE, wickets fallen ≥ 7 | DEFEND |
| MIDDLE, new batter (balls < 8) | DEFEND |
| MIDDLE, 2nd innings, RRR > 10 | ATTACK |
| Default | ROTATE |

### Bowler Intent

| Situation | Intent |
|---|---|
| POWERPLAY | CONTAIN |
| DEATH | ATTACK |
| New batter at crease | ATTACK |
| Default | CONTAIN |

---

## Consequences

### Positive

- Simulation is explainable: "batter attacked because death overs, RRR was 13"
- Naturally produces realistic phase-aware cricket behaviour
- Intent resolvers are independently testable
- New intent types (SLOG, GRIND, etc.) can be added without touching probability logic
- Captain AI can override intent in future sprints
- Player attributes can influence intent thresholds in future sprints

### Negative

- Adds a resolution step per delivery (negligible performance impact)
- Intent rules are currently hard-coded; future sprints should make them attribute-driven

---

## Alternatives Considered

### Direct Phase Modifiers (no Intent layer)

Apply different distribution adjustments based on innings phase alone, without an intent concept.

**Rejected.**

This is less extensible. It cannot model the same phase producing different behaviours based on match situation (e.g., death overs when winning vs. needing 60 from 18). It also removes explainability.

### Random Intent Assignment

Assign intent randomly with phase-weighted probabilities.

**Rejected.**

Produces noise rather than intelligence. A batter should always defend under pressure; random assignment breaks this.

---

## Future Extensions

- Attribute-driven intent thresholds (aggressive batters attack earlier)
- Captain intent override
- Partnership-driven confidence (long partnership shifts intent toward ATTACK)
- Pressure index modulating intent
- Player trait system (e.g., Finisher trait → prefers ATTACK in death overs)
