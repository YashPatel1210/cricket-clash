# Cricket Clash Knowledge Platform

## Vision

Cricket Clash is designed as two complementary systems:

1. A **Knowledge Platform** responsible for collecting, storing and analysing
   cricket information.

2. A **Simulation Engine** responsible for producing realistic cricket matches
   using knowledge generated by the platform.

The Simulation Engine never analyses historical data.

The Knowledge Platform never simulates matches.

Both systems evolve independently.

---

# Core Philosophy

The platform separates cricket information into three distinct layers.

```
Facts
        ↓
Knowledge
        ↓
Runtime Snapshot
```

Every layer has a different responsibility.

---

# Facts

Facts are objective observations collected from real cricket matches.

Examples:

- Player scored 78 runs.
- Match played at MCG.
- Strike rate was 164.2.
- Bowled by left-arm fast bowler.
- Match played under lights.

Facts never contain opinions or ratings.

Facts are immutable.

Facts are stored permanently.

Examples of future entities:

- Player
- Match
- Innings
- Delivery
- Venue
- Competition
- Team

---

# Knowledge

Knowledge is generated by analysing facts.

Knowledge is never entered manually.

Examples:

- Aggression
- Consistency
- Spin Ability
- Pace Ability
- Death Overs Skill
- Pressure Rating
- Captaincy Rating
- Fielding Position Rating

Knowledge can change whenever new matches are imported.

Knowledge represents the current understanding of a player, venue or team.

---

# Runtime Snapshot

Before every simulated match, the Knowledge Platform creates immutable
snapshots.

Snapshots combine historical knowledge with current match conditions.

Example inputs:

- Recent form
- Pitch
- Weather
- Venue
- Opposition
- Tournament
- Match importance
- Fatigue

Example output:

PlayerSnapshot

Snapshots never change during a match.

---

# Bounded Contexts

```
Knowledge Platform
│
├── Player Knowledge
├── Team Knowledge
├── Venue Knowledge
├── Competition Knowledge
├── Historical Matches
└── Analytics
```

```
Simulation Engine
│
├── Match Engine
├── Delivery Generator
├── Draft Engine
├── Benchmark
└── Validation
```

The Simulation Engine consumes snapshots but never accesses the database.

---

# Data Flow

```
Historical Matches
        │
        ▼
Database
        │
        ▼
Analytics
        │
        ▼
Knowledge
        │
        ▼
Snapshot Generator
        │
        ▼
Simulation Engine
```

Every component has a single responsibility.

---

# Player Evolution

Player represents identity only.

Examples:

- Name
- Country
- Batting Style
- Bowling Style
- Role

Player does not contain derived intelligence.

Knowledge about a player belongs to PlayerKnowledge.

Runtime information belongs to PlayerSnapshot.

---

# Guiding Principles

## Separation of Concerns

Simulation never performs analytics.

Analytics never performs simulation.

---

## Facts First

Everything begins with factual cricket data.

Knowledge is always derived.

---

## Immutable Runtime

Simulation uses immutable snapshots.

No runtime object modifies historical knowledge.

---

## Replace Knowledge, Not Code

Weekly ICC updates should regenerate knowledge.

Simulation behaviour improves without modifying simulation code.

---

# Long-Term Vision

The Knowledge Platform should eventually support:

- Weekly ICC updates
- Historical match imports
- AI player ratings
- Venue intelligence
- Captain intelligence
- Tournament balancing
- Fantasy cricket recommendations
- Match prediction
- Commentary generation
- Coaching analysis

The Simulation Engine remains independent from all data sources.

Only immutable snapshots cross the boundary.
